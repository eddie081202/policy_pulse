Scoring Engine
Judge Scoring Model

Four-Axis Policy Score

Every policy is evaluated across four independent dimensions. Your preference mode shifts the weighting to match your priorities.

💰 Price Score0 – 1.0

How competitive is the premium relative to market comparators for equivalent coverage?

📊 Policy Utilization Score0 – 1.0

What fraction of covered services did your billing actually utilise?

🛡️ Coverage Score0 – 1.0

Breadth and depth of coverage — deductibles, limits, exclusion density.

⭐ Relative Policy Quality0 – 1.0

How does this policy rank against semantically-similar alternatives in the vault?

Architecture
System Architecture

Full-Stack AI Platform

Three layers working in concert: a Next.js frontend, a FastAPI orchestration backend, and a multi-agent AI core backed by vector retrieval.

Frontend Layer
Next.js 14 (App Router)

React + TypeScript dashboard. File upload, real-time process stage tracking, result display with alternatives.

Tailwind CSS + Space Grotesk

Dark fintech design system. Glassmorphism cards, emerald accent palette, responsive grid layout.

API Route Proxy

Next.js API routes serve dataset files and proxy calls to the FastAPI backend securely.

Backend Layer
FastAPI + Uvicorn

Python async REST API. Receives multipart document uploads and orchestrates the full agent pipeline.

/api/judge/evaluate

Single endpoint that accepts contract + bill, runs three agents, and returns a structured JudgeResult payload.

Service / Entity Pattern

Clean separation: entities hold data models, services contain business logic. Each agent module is independently testable.

AI Core
Agent: Doc Reader

LLM-powered contract parser. Handles multi-modal input (text + image). Outputs structured coverage entity.

Agent: Bill Reader

Invoice and bill image understanding. Line-item extraction and coverage applicability mapping.

ChromaDB Vector Store

Persistent embeddings of real policy PDFs. Semantic nearest-neighbour search powers alternative retrieval.
